# Fable 5が使えないなら、Opus 4.8に“作業手順”を履かせればいい

Fable 5を使いたい。

でも、いつでも好きなだけ使えるわけではありません。
使用制限もあるし、従量課金ならコストも気になります。

では、Fable 5が使えないときはどうするのか。

今回私は、Opus 4.8をFable 5と偽るのではなく、「Fable 5に任せたいような作業手順」をClaude Skillとして作ることにしました。

作ったのは **opus-fable-workflow** というSkillです。
目的は、モデルの性能を変えることではありません。
作業を分解し、前提を整理し、最後に自己レビューすることで、Opus 4.8でも重要作業を安定して処理しやすくすることです。

## Fable 5を使いたかった理由

私は普段、Claude Codeでプログラミング、製造業の技術対応、noteやQuoraの記事作成をしています。

このうち「設計方針の相談」「複数ファイルをまたぐコードレビュー」「炎上リスクのある文章のチェック」のような仕事は、モデルの格がそのまま結果に出ます。
前提の見落としが一つあるだけで、レビューなら重大バグの見逃しに、文章なら誤解や炎上につながるからです。

だからこの種の作業は、一番強いモデルであるFable 5に任せたい。

## でも、いつでも使えるわけではない

Fable 5には使用制限があります。
上限に当たれば、その日はもう使えません。
従量課金で使うなら、今度はコストが気になります。

一方で、仕事は待ってくれません。
「Fable 5の枠が回復するまでレビューは保留」というわけにはいかない日があります。

## 性能ではなく、手順をSkillにする

そこで発想を変えました。

Fable 5が優れているのは事実ですが、重要作業で結果に効いているのは、モデルの地力だけではありません。
前提を確認してから作業する、作業を分解して一歩ずつ進める、出す前に見直す。
こうした「仕事の進め方」の部分は、モデルではなく手順の問題です。

であれば、手順のほうをClaude Skillとして固定してしまえば、Opus 4.8でもFable 5に任せたいような作業の失敗率を下げられるはずです。

例えるなら、高級スーツを買うのではなく、作業手順書を履かせて戦闘力を上げる方向です。

## 作ったもの：opus-fable-workflow

Skillの構成はこうなっています。

```
opus-fable-workflow/
  SKILL.md      … 発動条件、作業手順、出力ルール、禁止事項
  README.md     … 概要と使い方
  examples/     … 用途別テンプレート（Claude Code作業、記事作成、
                  技術調査、コードレビュー、メール、発動テスト）
```

リポジトリはこちらです。

https://github.com/garyohosu/skills

## このSkillでやること

発動すると、どんな依頼でも次の流れで処理します。

1. **タスク分類**：軽作業から高リスク作業までの重要度を判定する
2. **前提整理**：目的、対象、制約、不明点、仮定、リスクを整理する。不明点があっても、仮定を明示して先へ進む
3. **作業分解**：複雑な作業を3〜7ステップに分ける
4. **実行**：コードレビューなら重大バグから順に、記事なら読者像から、と用途別のチェック観点で成果物を作る
5. **自己レビュー**：出す前に、目的とのずれ、見落とし、断定しすぎ、日本語の自然さを確認する

ポイントは、この流れを毎回ユーザーに見せないことです。
途中経過を長々と出すとトークンを浪費するので、ユーザーに見せるのは結論、成果物、注意点だけに絞っています。

## 発動しすぎないようにした工夫

この手のSkillで一番怖いのは、誤字修正のような軽作業にまで発動して、トークンと時間を浪費することです。

そこで発動条件を二段構えにしました。

- 重要作業では発動する：設計相談、複数ファイルのレビュー、バグ原因の推定、重要メール、炎上チェックなど
- 軽作業では発動しない：短い翻訳、誤字修正、短いメールの軽い校正、雑談など

ただし例外を一つ置いています。
軽作業でも、ユーザーが「重要」「丁寧に」と言ったら発動する、というルールです。
短い翻訳でも、契約文なら丁寧にやってほしい場面があるからです。

## 実際の発動テスト結果

作りっぱなしにせず、新しいセッションを6本立てて発動テストをしました。

まず、発動してほしい3件。

- 「Fable-likeモードで手順書をレビューして」 → 発動した
- 「Opus 4.8で丁寧に。論理構成を見直して」 → 発動した
- 「重要なレビューです。仕様の矛盾を洗い出して」 → 発動した

次に、発動してほしくない3件。

- 「納品書って英語でなんて言う？」 → 発動しなかった
- 「この一文の誤字だけ直して」 → 発動しなかった
- 「雑談だけど、プログラマーあるあるを教えて」 → 発動しなかった

6件とも期待どおりでした。
非発動側の出力は32〜586トークンで済んでおり、軽作業が軽作業のまま処理されていることも確認できました。

最初からこうだったわけではありません。
初版はモード名の「Fable-like」がSkillの説明文に入っておらず、そのままだと取りこぼすところでした。
発動条件は書いて終わりではなく、テスト用のプロンプト集（examples/trigger-check.md）を用意して確認する。
ここは普通のソフトウェア開発と同じでした。

## 運用ルール

Skillを増やしていくと、管理の仕組みのほうが大事になってきます。
私は次のルールで運用しています。

- 正本はGitHubの skills リポジトリに置く
- `~/.claude/skills` は同期先。リポジトリから robocopy のミラーで反映するので、直接編集しない
- SKILL.md の description は1行で書く

最後のルールは今回の開発でハマった点です。
descriptionをYAMLの折りたたみ形式（`>-`）で複数行に書いたところ、README自動生成スクリプトが読めず、スキル一覧に「>-」とだけ表示されました。
自作の自動化スクリプトがある環境では、書式の自由度はスクリプト側の都合で決まります。

## まとめ

- モデルの性能差は、作業手順の固定である程度カバーできる
- 具体的には、タスク分類、前提整理、作業分解、自己レビューをSkillとして強制する
- 発動条件は「重要作業で発動、軽作業で非発動、ただし指定があれば発動」の二段構えにして、テストで確認する

最後に一つだけ、はっきり書いておきます。

このSkillを使っても、Opus 4.8はFable 5と同等にはなりません。
やっているのは「Opus 4.8をFable 5と偽る」ことではなく、「Fable 5に頼みたいような作業を、Opus 4.8でも失敗しにくくする」ことです。

その線引きさえ守れば、上位モデルが使えない日でも、重要な仕事をそれなりの品質で回せます。
同じように「上のモデルが使えたらなあ」と思っている方は、性能ではなく手順をSkillにする方向を試してみてください。
