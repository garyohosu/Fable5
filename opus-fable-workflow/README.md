# opus-fable-workflow

Fable 5が使えない環境でも、Opus 4.8で高品質な推論・文章作成・コードレビュー・技術調査・設計相談を行うための作業手順スキルです。

> **このSkillはOpus 4.8をFable 5と同等にするものではありません。Fable 5向きの重要作業を、Opus 4.8でも安定して処理しやすくするための作業手順Skillです。**

## 概要

Fable 5に任せたいような重要タスクを、Opus 4.8でもできるだけ安定して処理できるように、以下を標準手順として実行します。

- タスクの重要度判定と分類
- 作業の段階的分解（3〜7ステップ）
- 前提条件・不明点・リスクの整理
- 確認 → 設計 → 実行 → 見直しの流れでの処理
- 長文コンテキストの圧縮
- 最終出力前の自己レビュー

## 何ができるか

- 難しい技術相談、バグ原因の推定、設計方針の相談
- 複数ファイルをまたぐコードレビュー
- Note・Quora向けの記事作成、長文資料の要約と分析
- 重要な仕事メールの文面作成、炎上リスクのある投稿文の論理チェック
- Claude Code、Codex、Gemini CLIなどAIツールの使い分け・運用相談
- 仕様書、手順書、README、ブログ記事の改善
- 作業内容に応じたモデル使い分け（Fable 5 / Opus 4.8 / 安いモデル）の提案

## 何ができないか

- Opus 4.8をFable 5そのものに変えることはできません
- モデルの素の性能（推論力・知識量）を引き上げることはできません
- Fable 5と完全に同じ品質を保証することはできません

## Fable 5との違い

Fable 5はモデル自体の性能で高品質な出力を出します。
このスキルは、Opus 4.8に対して「作業手順の工夫」で品質を底上げするものです。

- Fable 5：素の推論力で複雑なタスクを一度に処理できる
- Opus 4.8 + このスキル：タスクを分解し、前提を確認し、見直しを挟むことで、見落としや論理の飛躍を減らす

「Fable 5と同等」ではなく「Fable 5風の作業手順」です。

## Opus 4.8で使う意味

- 重要ポイントの見落としを減らせる
- 長文でも構造化して扱える
- 複雑な依頼を段階的に処理できる
- 最終回答前の見直しで品質が安定する
- 無駄なトークン消費を減らせる

## 使い方

1. このフォルダを `~/.claude/skills/opus-fable-workflow/` に配置する（Claude Code の場合）
2. 重要な作業を依頼するとき、以下のように言う
   - 「Fable 5がないので、Opus 4.8で丁寧にやって」
   - 「このコードをしっかりレビューして」
   - 「Fable 5っぽく作業して」「Fable-likeモードで」
3. または `examples/` のテンプレートをコピーして依頼文を作る

発動しやすいフレーズ・発動しないケースの一覧は `examples/trigger-check.md` を参照してください。
軽作業でもこのスキルを使わせたい場合は「丁寧に」「重要」を付けてください。

## 例

```
Opus 4.8 Fable-like Modeでコードレビューしてください。

対象：
src/parser.py と src/validator.py

確認してほしい観点：
- バグ
- 例外処理
- 境界条件
- 最小修正案

条件：
- 重大な問題から順に指摘する
- 不要なリファクタリングは避ける
- 日本語で説明する
```

その他の例は `examples/` を参照してください。

- `examples/claude-code.md` — Claude Code作業
- `examples/writing.md` — 文章・記事作成
- `examples/technical-research.md` — 技術調査
- `examples/code-review.md` — コードレビュー
- `examples/email.md` — メール作成

## 注意点

- 短い翻訳、誤字修正、雑談などの軽作業にはこのスキルは使いません（過剰な手順はトークンの無駄になるため）
- 不明点がある場合、質問せずに進められるときは仮定を明示して続行します。仮定が間違っていたら指摘してください
- 重要な判断では不確実性を明示します。断定がないのは仕様です

## 禁止事項

このスキルでは以下を禁止しています。

- Opus 4.8をFable 5と偽ること
- Fable 5と完全に同じ性能だと説明すること
- 根拠のない断定、不明点の勝手な事実化
- 長すぎる内部推論の出力
- ユーザーの意図と違う大規模変更
- 勝手なファイル削除、勝手なコミット、勝手なpush
- 英語出力が不要な場面で英語だけ出すこと
