# 発動確認用プロンプト集

このスキルの発動精度を確認するためのテストプロンプト。
新しいセッションでそれぞれ入力し、スキルが使われるかどうかを確認する。

## 発動すべきプロンプト（should trigger）

1. 「Fable 5がないので、Opus 4.8でこの設計方針を丁寧に検討して」
2. 「Fable-likeモードで src/ 以下のコードをレビューして」
3. 「Fable 5っぽくしっかり見てほしい。装置ログのCSVが夜間だけ欠損する原因を推定して」
4. 「重要なレビューです。この仕様書の矛盾を洗い出してください」
5. 「これは複雑作業なので段階的に進めて。測定スクリプトの改修と手順書の更新をまとめてやりたい」
6. 「Note向けにClaude CodeとCodexの使い分け記事を書きたい。構成から丁寧に」
7. 「取引先への納期遅延メール、重要なので失礼のない文面をしっかり作って」
8. 「このX投稿、炎上しないか論理チェックして」
9. 「upload.py と filename.py をまたぐ文字化けバグの原因を調べて直す方針を出して」
10. 「Opus 4.8で丁寧にやってほしい。README全体の論理構成を見直して」

## 発動すべきでないプロンプト（should NOT trigger）

1. 「"納品書" って英語でなんて言う？」
2. 「この一文の誤字だけ直して：計測データを保存しまス」
3. 「さっきのメール、"よろしくお願いします" を "よろしくお願いいたします" に変えるだけでいい？」
4. 「ESP32って技適通ってるの？」（すぐ答えられる一般知識）
5. 「この文を少し柔らかく言い換えて：至急対応してください」
6. 「今日は暑いね。作業の合間の雑談だけど、最近のAIニュースで面白いのあった？」
7. 「変数名 tmp を temperature に一括置換して」
8. 「この箇条書き3つ、順番だけ入れ替えて」

## 境界ケース（指定語があれば発動に倒れる）

1. 「短い翻訳だけど重要な契約文なので丁寧に：{英文}」 → 発動する（「重要」「丁寧に」指定あり）
2. 「一行のコミットメッセージ、Fable 5っぽく考えて」 → 発動する（「Fable 5っぽく」指定あり）

## 判定のしかた

- 発動した場合：回答の冒頭または処理の流れに「目的の再定義 → 前提整理 → 段階的実行 → 自己レビュー」の構造が現れる
- 発動しない場合：直接短く回答される
- should-trigger で発動しない例が続く場合は、SKILL.md の description のトリガー語を強める
- should-NOT-trigger で発動してしまう場合は、除外リスト（短い翻訳・誤字修正・雑談など）に具体例を追加する
